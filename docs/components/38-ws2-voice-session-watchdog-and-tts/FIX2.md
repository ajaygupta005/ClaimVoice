## Fix 2: Real Voice Output for Claude-Composed Answers

## Goal

Make the assistant audibly respond in a high-quality AI voice after Claude composes the answer.

The user experience should feel like:

```text
user speaks
-> browser STT captures the question
-> LangGraph/tools/Claude produce the answer text
-> TTS speaks that answer aloud
-> UI returns to Ready
```

## Important Clarification

Claude is the text reasoning and answer-composition layer.

Claude is not the voice layer in this application unless a supported Anthropic voice/audio API is explicitly available and configured.

Therefore the correct product behavior is:

```text
Claude answer text + Google/Browser/other TTS voice
```

Do not label the audio as “Claude voice” unless the audio is actually generated by an Anthropic-supported voice API.

Recommended UI wording:

- `Claude answer`
- `Voice: Google TTS`
- `Voice: Browser fallback`

Not recommended:

- `Claude voice`
- `Claude speaking`

## Current Problem

The assistant shows Claude-generated text, but the user does not hear a natural voice response.

Current weak points:

- answer text is added to transcript but not reliably spoken aloud
- TTS status may stay inactive
- the UI can get stuck after answer generation
- there is no reliable audio provider contract
- there is no voice selection
- browser fallback behavior is unclear

## Required Behavior

### 1. TTS Provider Selection

The system must support provider selection through configuration.

Required config:

```text
VOICE_AGENT_TTS_PROVIDER=browser|google
GOOGLE_TTS_LANGUAGE_CODE=en-US
GOOGLE_TTS_VOICE_NAME=<configured-google-voice>
```

Default:

```text
VOICE_AGENT_TTS_PROVIDER=browser
```

If Google TTS is not configured, the UI must still speak using browser `speechSynthesis`.

### 2. Google TTS First-Class Path

When `VOICE_AGENT_TTS_PROVIDER=google`:

- backend receives assistant answer text
- backend calls Google Cloud Text-to-Speech
- backend returns playable audio to the browser
- browser plays the audio
- UI shows `Voice: Google TTS`

Preferred voice family:

- Google Chirp / Chirp 3 HD voice where available

The exact voice name should be configurable and not hardcoded in UI code.

### 3. Browser TTS Fallback

When Google TTS is unavailable or not configured:

- use `window.speechSynthesis`
- pick the best available English voice
- allow a configured preferred browser voice name if present
- show `Voice: Browser fallback`

Fallback is acceptable for demo reliability, but it must be visually labeled as fallback.

### 4. Audio Contract

Backend TTS endpoint should expose a simple synthesis contract.

Request:

```json
{
  "text": "Your in-network urgent care copay is $75 per visit.",
  "voice": "default",
  "format": "mp3"
}
```

Response when audio is available:

```json
{
  "ok": true,
  "provider": "google",
  "voiceName": "configured voice name",
  "mimeType": "audio/mpeg",
  "audioBase64": "..."
}
```

Response when provider is unavailable:

```json
{
  "ok": false,
  "provider": "google",
  "reason": "not_configured",
  "fallback": "browser"
}
```

The frontend must never receive or expose Google credentials.

### 5. Speak-Back Lifecycle

After an assistant answer is received:

1. append answer to transcript
2. enter `speaking`
3. synthesize or fallback to browser voice
4. play audio
5. return to `ready` when playback ends

If playback fails:

- cancel audio
- show recoverable TTS error
- return to `ready`
- do not block the next turn

### 6. Interrupt Behavior

The user must be able to interrupt voice output.

If user taps mic while the assistant is speaking:

- stop current audio playback
- cancel browser speech synthesis if active
- clear TTS timers
- start a new listening turn

This must work even if Google audio playback or browser speech synthesis is mid-sentence.

### 7. TTS Watchdog

TTS must not trap the UI.

Required timeout:

```text
maxPlaybackMs = estimatedSpeechDurationMs + 5000
```

If playback does not finish by timeout:

- stop audio
- mark TTS as timed out
- return to `ready`

### 8. Connection Status

Connection status must show the voice layer separately from Claude.

Examples:

```text
Claude: connected
Voice: Google TTS
```

or

```text
Claude: connected
Voice: Browser fallback
```

or

```text
Claude: connected
Voice: unavailable
```

Claude status must come from backend `composer_mode=claude`.

Voice status must come from the selected TTS provider and playback result.

### 9. UI Copy

Use wording that does not imply Claude itself has spoken audio unless that is truly implemented.

Recommended:

- `Claude composed the answer`
- `Speaking with Google voice`
- `Speaking with browser voice`
- `Voice unavailable`

Avoid:

- `Claude voice`
- `Claude is speaking`
- `Anthropic realtime voice`

## Acceptance Criteria

- Claude-composed answer is spoken aloud after it appears.
- UI clearly distinguishes `Claude answer` from `Voice provider`.
- Google TTS is used when configured.
- Browser TTS fallback works without Google credentials.
- User can interrupt speech by tapping mic.
- UI returns to `Ready` after speech ends.
- UI returns to `Ready` after TTS failure.
- TTS timeout cannot leave the app stuck.
- No API keys are exposed to the browser.
- The app does not claim to use “Claude voice” unless an actual supported Claude voice API is wired.

## Out of Scope

- Real Twilio phone audio return
- Streaming partial TTS chunks
- Voice cloning
- Custom branded voice training
- Letting Claude choose voice style dynamically
- Replacing the LangGraph pipeline
